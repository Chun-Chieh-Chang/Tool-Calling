### Agent 技能開發標準化規範建議書：構建高精確度的技能檢索體系

##### 1\. 核心設計哲學：從提示工程轉向檢索工程

當 Agent 系統的技能 (Skills) 規模從單位數增長至百倍級別時，傳統的「平鋪式」Prompt 注入將面臨嚴峻的技術瓶頸。開發者常觀察到模型出現「路徑選擇失效」或在高維度選擇下的隨機擾動現象（Model Stochasticity under High-Dimensional Selection，俗稱「抽風」）。即便系統存在完全匹配的技能，模型也可能因注意力分散而拒絕呼叫，或誤選語義相近但功能錯誤的技能。這並非模型推理能力的退化，而是當技能規模化後，技能路由（Skill Routing）的本質已從邏輯推理演變為一個複雜的 **檢索工程問題** 。為了解決大規模 Agent 的路由精度挑戰，本規範採納 Anthropic 的核心戰略思想： **「漸進式加載 (Progressive Disclosure)」** 。**漸進式加載 (Progressive Disclosure)**  系統在初始路由階段嚴禁將所有技能的完整 Prompt 注入上下文。相反，應僅優先讀取技能的「名稱 (Name)」與「摘要 (Description)」。唯有當路由模型確認選用特定技能後，才動態加載該技能的完整執行邏輯與參數 Schema。**策略轉向：從內部優化到外部描述**在大規模場景下，工程資源應從「技能內部的 Prompt 微調」轉向「外部檢索特徵的工程化」：

* **傳統模式：**  專注於技能內部的指令細節，試圖透過 Prompt 解決所有邊界問題。  
* **檢索優先模式：**  優化技能的語義標籤，確保在向量空間中具備極高的辨識度，減少語義碰撞。在確立檢索優先思維後，首要任務是定義具備高辨識度的技能標籤，這是提升路由命中率的基石。

##### 2\. 語義匹配優化：技能名稱與摘要的標準化規範

在向量檢索邏輯中，技能名稱扮演「標題」角色，摘要則是「內容提要」。模糊的命名（如「數據分析」）會導致不同技能在向量空間中的距離過近，產生 **語義碰撞 (Semantic Collision)**  與 **召回雜訊 (Recall Noise)** 。本規範要求必須最大化技能向量之間的  **Cosine Similarity 距離** 。**命名禁令與標準對比表**| 類別 | 模糊命名（禁止使用） | 高辨識度命名（標準規範） || \------ | \------ | \------ || 辦公類 | 通用辦公 | 企業會議紀要生成與行動項提取 || 分析類 | 數據分析手 | 銷售數據趨勢分析與視覺化報表生成 || 開發類 | 代碼處理專家 | React 前端組件 UI 代碼自動生成 |  
**標準化描述生成指令 (Meta-Prompt)**本規範強制要求使用以下 Meta-Prompt 指引 LLM 生成技能描述，以確保「語義獨立性」：**Meta-Prompt Template**  請為名為 技能名稱 的功能撰寫標準化描述。要求：

1. 字數限制：50-80 字。  
2. 使用主動動詞開頭（如：生成、提取、分析）。  
3. 確保「語義獨立性」，明確界定與同類技能的區隔。  
4. 嚴禁包含具體實現邏輯，僅描述其唯一核心價值。清晰的名稱與描述僅解決了靜態屬性，必須引入動態觸發場景，以提升模型對用戶意圖的感知識別。

##### 3\. 「觸發場景 (Trigger Scenarios)」：從功能描述轉向情境驅動

模型執行路由決策的本質是\*\*「意圖與情境的語義匹配 (Intent-to-Context Matching)」\*\*。因此，描述「何時使用 (When to use)」的命中率提升效果遠高於描述「它是什麼 (What it is)」。**觸發場景撰寫規範**所有技能描述必須包含具體的使用情境，確保與用戶可能的提問方式（User Intents）高度契合。

* **格式規範：**  必須以條列式列出至少 3 個具體場景。  
* **範例（PPT 生成技能）：**  
* 當用戶明確要求製作「匯報材料」或「投影片」時。  
* 當用戶提到需要準備「融資路演」或「季度總結」的視覺化展示時。  
* 當用戶尋求創建「演示文稿」或「簡報 PPT」的結構化內容時。**影響分析 (So What?)**精確的場景描述能直接降低路由階段的 **認知負荷 (Cognitive Load)** 。透過將「功能」映射至「場景」，模型能更直觀地建立用戶請求與技能間的關聯，減少因語義模糊產生的決策猶豫。為了進一步收斂路由邊界，除了定義「正向觸發」，更必須明確定義「負向邊界」。

##### 4\. 負樣本描述 (Negative Samples)：明確「不適用範圍」

負樣本描述（When NOT to use）是區分工業級規範與一般開發的關鍵。在檢索系統中，負樣本充當  **"Hard Negatives"** ，能有效防止技能被過度呼叫或誤觸發 (False Positive)。**負樣本編寫模板與範例**本規範要求將負樣本納入技能定義中，並建議將其作為 Few-Shot 路由提示詞的一部分。  
\[技能名稱\]: 前端代碼生成專家  
\[適用範圍\]: 專注於 HTML/CSS、JavaScript 及 React 組件的生成。  
\[負樣本描述 (When NOT to use / Hard Negatives)\]:  
\- 不適用於數據庫架構設計或後端 API 接口開發。  
\- 涉及 SQL 查詢語法編寫或性能調優時，嚴禁調用此技能。  
\- 當需求涉及服務器部署或後端邏輯架構時應排除。

**邊界劃分邏輯**透過排除法（Method of Exclusion）優化檢索精度。當用戶需求處於灰色地帶時，明確的負樣本描述會強制路由系統去尋找更精確的專業技能，而非落入看似相關但實則不符的路徑。當單體技能描述標準化後，需透過結構化組織應對全球規模的技能池。

##### 5\. 技能層級化架構 (Skill Tree)：縮小路由搜索空間

將上百個技能全量平鋪會導致模型在單次判斷中面臨過大的搜索空間。本規範強制採用\*\*「層級化路由 (Hierarchical Routing)」\*\*架構，將系統從一次性匹配轉變為多階段決策。**分層結構設計規範**

* **第一層：分類閘口 (Classification Gate)**  
* 由  **Router Agent**  擔任，負責將需求歸類至頂級類別（如：研發、市場、財務、運營）。  
* **第二層：精確檢索 (Skill Retrieval)**  
* 在鎖定的子樹空間內，進行具體技能的檢索與匹配。**典型技能樹結構示範：**  
* **研發 (R\&D)**  
* 代碼生成  
* 代碼評審 (Code Review)  
* 測試案例自動生成  
* **市場 (Marketing)**  
* 文案創意生成  
* 社交媒體趨勢分析  
* 競品數據抓取這種架構確保了模型每次只需在受限的（\<10 個）選項中做選擇，顯著提升了系統穩定性。

##### 6\. 技術實現路徑：召回與重排 (Recall & Rerank)

當技能數量達規模化級別，Agent 路由系統應完全對齊 RAG（檢索增強生成）的設計思路，引入雙層檢索機制。**兩階段路由標準化流程**

1. **檢索階段 (Recall)：**  使用 Embedding 向量計算或關鍵字檢索，從海量技能池中初步過濾出 Top 5-10 個候選技能。此階段可 **減少約 80% 的 Context Window 消耗** 。  
2. **決策階段 (Rerank)：**  將候選技能的詳細描述與負樣本注入 LLM，進行最終的推理判定，確定執行技能。**評估指標建議 (KPIs)**  
* **命中率 (Hit Rate)：**  請求正確路由至目標技能的比例（工業級目標 \> 95%）。  
* **誤觸率 (False Positive Rate)：**  技能在不適用場景下被錯誤觸發的機率。  
* **路由延遲 (Routing Latency)：**  從輸入到確定技能的端到端耗時（透過分層與召回機制優化）。

##### 結語

標準化的技能開發規範是構建可擴展 Agent 生態系統的基石。透過「漸進式加載」與「層級化路由」，開發者能有效解決模型在大規模場景下的決策失效問題。本規範不僅優化了目前的路由精度，更為未來支持成千上萬個技能的 Agent 集群奠定了架構基礎。
